1執行摘要(30 秒版)

這個工具做一件事:把外部給的「鏡頭掃描視差序列」解成一張 8×6 的 DCC 係數表,驗證後打包成 993 bytes 的 EEPROM 校正 block。DCC(Defocus Conversion Coefficient)是 PDAF 對焦的心臟——runtime 拿它把「相位視差(pixel)」直接換算成「對焦馬達要走幾步(DAC)」。

核心算法一句話 每一區把 N 個掃描點做線性回歸 DAC = k·disparity + b:斜率 k 就是 DCC(要燒錄的值),截距 b 是合焦點(拿去跟 focus 曲線峰值互相對質,兩者吵架超過容差就判 FAIL)。數學上只是最小平方閉式解,巧思在於工程防護:單位契約、EIV 偏差對策、粒度聚合、失敗落盤。
  • 解的規模:預設 10 幀 × 48 區(8×6)= 480 個樣本點 → 48 條回歸線 → 48 個 DCC 值。
  • 介面彈性:輸入粒度可以不是 8×6(例如 144×108,自動聚合)、單位可以是兩種(自動轉換)、掃描點數 5..50、fitter 四選一——全部 config 可調,不改程式。
  • 品質現況:95/95 測試綠、離線重算 bit-exact、數量閉環驗算文化(993 bytes、480 樣本都算得攏)。
  • 定位:離線 + 實驗工具(2026-07-17 決議),不碰硬體;實機擷取、SAD 計算、實體燒錄都在牆外。

3核心求解算法:兩條擬合 + 一個交叉驗證

3.1 主擬合:逐區線性回歸,斜率就是答案

物理背景很直觀:鏡頭從遠掃到近(FAR→NEAR,10 個 DAC 位置),每個位置量到一個相位視差。視差與離焦量成正比,所以 DAC 對視差畫出來就是一條直線:

DAC = k · disparity + bk = DCC [DAC/raw_pixel],b = 合焦點 [DAC]

解法是標準 OLS 閉式解(中心化後 k = Sxy/Sxx),每區獨立、無迭代、確定性——這是「離線重算 bit-exact」能成立的基礎。防護條件:有效樣本 < 8 → E-D03;解出 k ≤ 0 → E-E01(物理上 DCC 恆正,k 變負幾乎都是 L/R 影像對調,錯誤訊息會直接提示查這個);r² < 0.98 出警告。

3.2 進階:量測噪聲在 x 軸上——EIV 問題與四種 fitter

這裡有一個 reviewer 必須點出的統計細節:回歸式裡 DAC 是命令值(精確),disparity 是量測值(含噪)——含噪的是自變數,這叫 errors-in-variables(EIV)。教科書後果:前向 OLS 的斜率有衰減偏差(往 0 拉,DCC 系統性偏低),且偏差隨噪聲 σ² 放大。M2 對此做了完整的實驗閉環(50 seeds 蒙地卡羅,配對比較):

fitter做法偏差變異(CV)綜合 RMSE建議場景
ols_forward對 disparity 回歸(v1 預設)衰減偏差,σ=0.5 時僅 −0.017%最低之一次佳σ ≤ 0.5(典型產線):維持預設
ols_inverse反向擬合後取倒數(消偏)接近無偏最差四者最差不建議單獨使用
wls_inverse反向 + quality 加權 w=qγ接近無偏最低全部 σ 下最佳σ ≥ 1(低品質 SAD):啟用,γ=1 即可
deming廣義 EIV(δ 參數)δ=0 與 inverse 位元級一致(UT 釘死),作為理論橋接保留研究/對照用
  • 實驗量級:前向偏差 σ=0.5 → −0.017%(埋在雜訊底)、σ=2.0 → −2.30%(顯著);容差門檻是 10~20%,所以典型噪聲下前向偏差可忽略,不換預設是有數據支撐的決策,不是偷懶。
  • WLS 對 γ 誤標不敏感(γ=0.5 和 2 的差距遠小於用不用 WLS 的 ~9% CV 改善)——整合者不必精確標定 γ 也能安全啟用。
  • 實料驗證:TEST0720 那筆 σ=1.29 的 20 點序列,forward vs wls 中央 DCC 差 +0.54%,方向與量級都吻合理論預測。

3.3 交叉驗證:用一條「獨立證據鏈」對質

回歸截距 b 說「合焦點在這裡」;但 focus 曲線(每區逐幀的清晰度量值)的峰值也說了一次「合焦點在這裡」。兩個來源物理上該一致,不一致就是有事——chart 距離擺錯、場曲、或序列本身有問題:

err = |intercept − focus_peak| / sweep_span逐區計算;span 由 config 推導(示例 480),禁止寫死

focus 峰值的求法:4 階多項式最小平方擬合(階數 config 可調),峰值必須嚴格落在掃描窗內縮一步的範圍,出界即 E-F01——出界最常見的原因正是 chart 距離擺錯,錯誤訊息會提示。這條驗證鏈的靈敏度後來也被反過來利用:M2 的 chart 距離公差換算工具(--chart-tol)就是沿這條鏈路把「治具要擺多準(±cm)」跟「DCC 誤差預算(%)」做成雙向換算(示範條件 nl=0.05 → ±1.851 cm)。

算法全貌的驗算閉環(鐵律 5) 10 幀 × 48 區 = 480 樣本 → 48 條回歸(48 個 k + 48 個 b + 48 個 r²)→ 48 個 focus 峰值 → 48 個 err → 1 個模組判定。每一步的數量都對得上,對不上就停——這個文化在專案裡抓過真 bug(PhaseD-1 勘誤)。

4輸入介面:disp_seq.json 與其彈性

唯一的主要輸入是外部 SAD 模組產出的 disp_seq.json(v1,2026-07-16 與外部團隊凍結)。設計上的聰明之處在於:凍結的是結構與驗證規則,把會變的東西全部參數化:

欄位內容彈性
unit / pitch_x"raw_pixel" 或 "pd_image_grid" + 節距驗算值活 兩種單位皆收,讀取端統一轉 raw_pixel(§5)
dacs [N]掃描 DAC 序列,嚴格遞增(FAR→NEAR)活 N = 5..50(config num_positions);死 方向不可反(VCM 磁滯,鐵律 3)
grid_w × grid_h外部 SAD 的輸出粒度活 可 ≠ 8×6(如 144×108 pattern block 級),工具自動聚合
data [N][h][w]disparity;null = 無效樣本活 cell 級無效標記,聚合與回歸自動跳過
focus [N][h][w]逐 cell 清晰度值(必填)活 量綱不拘(只用峰值位置),同序列內一致即可
quality(選填)SAD 曲線形狀導出的信心值活 有 → 加權聚合/WLS 權重;沒有 → 等權,功能不減

粒度聚合:讓外部團隊不用遷就我們的 8×6

這是整合彈性的關鍵一手。外部 SAD 的自然輸出粒度(例如每 pattern block 一值 → 144×108)跟 DCC 的 8×6 沒必要一樣,工具端以 cell 中心座標歸區後聚合:median(預設,抗離群)或 weighted_mean(quality 加權,最小變異);區內有效 cell 比例低於門檻(預設 0.5)→ 整區該幀記無效。換句話說,外部改粒度、改品質策略,我們只動 config。

守門實例:驗證規則真的在工作

實跑案例(2026-07-20,TEST0720) 拿一筆 GUI 產的 20 點序列配 10 點的 config 跑,讀取端立刻以 E-D01 — dacs 長度(20)!= num_positions(10) 中止,訊息直指原因;把 config 的 num_positions 改 20 後一次通過(PASS,五檔齊全)。錯誤碼有語意、修正路徑清楚——這是介面守門該有的樣子,整合對接時的除錯成本會低很多。

5單位契約:全案只有兩個轉換閥門

視差有兩種單位:raw_pixel(感光器像素)和 pd_image_grid(PD 影像格,相差 pitch_x = 16 倍)。16 倍搞錯一次,DCC 就錯 16 倍——這是本領域的經典事故源。專案的對策不是「小心一點」,而是架構上讓錯誤沒地方發生:

  • 稽核設計:report.json 同時記兩種單位的 DCC,並內建 dcc_ratio_check = dcc_pd_grid / dcc_raw_px,恆應等於 16——若 ≈1(漏乘)或 ≈256(重複乘)直接示警。測試 UT-09 把這兩種事故模式都釘死了。
  • 對整合者的意義:runtime PDLIB 要哪種單位還沒定案(開放問題 #2,延期),但 output_disparity_unit 雙路徑已備好——定案那天翻一個 config 值就完事,不動程式。

7彈性總覽:整合者不改程式能調什麼

原則(FR-01):所有 Phase 的可調參數都必須進 config schema,不得散落程式常數。下表按整合情境整理——左欄是「當外部條件變了」,右欄是「你只需要動哪個 config」:

噪聲變大(低品質 SAD / 稀疏取樣)regression.fitter = wls_inverse + weight_gamma依 fitter 實驗:σ≥1 才需要;γ 誤標不敏感

刻意「不彈性」的地方(這是優點) 掃描方向(VCM 磁滯)、core 內單位(raw_pixel 唯一)、依賴方向(gui/cli → app → io/hal → core)——這三處是介面上就無法違反的設計面保證,不是靠測試守。彈性給在該給的地方,契約鎖在會出事故的地方。

8Reviewer 意見:品質現況、風險與待辦
