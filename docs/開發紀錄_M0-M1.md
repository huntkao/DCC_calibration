## 2. 重大架構決議(時序)

1. **2026-07-11:前期離線化**。實機擷取、PD 抽取/gain/SAD disparity、EEPROM 燒錄
   全部外部化,本工具消費 `disp_seq.json`(SPEC-004 §3a)。介面(HAL/NvmIF)保留待 M2。
   → 這使開放問題 #1/#2/#4 全部不阻擋開發(延期至 M2)。
2. **2026-07-11:focus value 必須隨序列輸入**。像素處理外部化後,Phase F 的 fv 無法自算
   ——序列格式因此把 `focus[N][h][w]` 列必填。跨團隊溝通時務必強調。
3. **2026-07-12:實作棧 C++17 + Dear ImGui**(原規格 Python+NumPy,已勘誤 NFR-04/SPEC-003)。
   GUI = imgui v1.91.9b-docking + implot v0.17 + glfw 3.4(FetchContent 鎖版)。
   分層:`gui/cli → app → io/hal → core` 嚴格單向;**GUI 能做的 CLI 必須也能做**(解耦試金石)。
4. **2026-07-12:dcc_sim 升格獨立 library**。Sim 是與外部 SAD 模組平行的「生產者」,
   產出同格式序列;GUI 的 Sim 工作台/靈敏度掃描都以它為後端。
5. **TexInspect → 自製 RAW 檢視器**。ImGuiTexInspect 停更(2021-2023)且需對 1.91 打補丁;
   改用 ImPlot::PlotImage 自製(縮放平移、DN 探針、PD 疊層、8×6 區格聯動、像素 snap 描框)
   ——功能為其超集且零過時依賴。若日後需要其像素值矩陣 annotation 再評估。
6. **自製 Logger 取代 spdlog**(M1d)。需求(雙 sink:繁中人讀 .log + 結構化 .jsonl、
   thread-safe)以 ~60 行自有代碼滿足,少一個外部依賴;若 M2 需要 rotation/異步再換。

## 3. 重要發現(依影響排序)

### 3.1 規格勘誤類(M0 審查發現)
- **ratio_check 公式自相矛盾**(SPEC-004 §3 舊版):公式恆等於 1 卻寫「應等於 pitch_x²/pitch_x」。
  勘誤為 `dcc_pd_grid / dcc_raw_px == pitch_x`(≈1=漏乘、≈pitch²=重複乘)。UT-09 釘死。
- **合成真值矛盾**:SPEC-005 §2 合焦 420,但舊 UT-05/06 要求 b=421、峰值 427(數學上不可能)。
  勘誤為 b=peak=420;非零 err 情境改由注入建構。
- **E-F01 測試案例隨 margin 失效**:margin 0.1 使 sweep 變 180→660,「合焦 560」不再出界
  → 案例參數化為「合焦 > NEAR−step(示例 640)」。教訓:**測試準則不可寫死推導值**。

### 3.2 誤差預算推導(SPEC-005 §3a,關鍵數學)
- 斜率相對誤差 ≈ `σ_d·k/√Sxx`;NFR-02(CV<2%)為綁定條件 → σ_d ≤ 0.65(margin=0)/0.78(0.1)raw px。
- **舊 UT-03/04 容差(0.35×/0.25×pitch = 5.6/4 px)鬆了約一個數量級,已作廢**;
  改為對外部 SAD 模組的驗收條件:|bias| ≤ 0.3、σ ≤ 0.5 raw px(對 margin 全域保守)。
- EIV(自變數含噪)衰減偏差在預算內 ≈ −0.4% 可忽略 → **v1 用一般 OLS 有數學依據**;
  fitter 介面可抽換(SPEC-003 §5)。IT-07 蒙地卡羅(σ=0.5+bias=0.3×10 seeds → CV<2%)閉環驗證。
- 全 sweep disparity 動態範圍僅 ±19.3 raw px ≈ ±1.2 PD 影像格 → 外部 SAD 的次像素
  內插品質是全案精度第一決定因素(開發備忘 #1)。

### 3.3 物理/演算法洞察(M1c/M1d 實測)
- **理想線性模型下 DCC 對合焦偏移完全不敏感**(斜率不變,只動截距)。CLI `--scan` 實測
  nl=0 時 ±60 DAC 全程 ΔDCC=0.0000%。真實靈敏度來自視差曲線非線性:
  注入 nl 後 **靈敏度 ≈ 2·nl·Δ/240**(nl=0.05、Δ=+40 → 實測 +1.69% vs 理論 1.67%)。
  → 開放問題 #3 的判讀框架:量到實模組 nl 即可換算 chart 距離公差。
- **非線性模型修正(使用者物理直覺查核)**:原 nl 為偶次項(殘差 ∝ u²,兩端同向、鐘型),
  與手冊 canonical 的 S 型軌跡不符。修正為雙項模型
  `disp = (u/k)·(1 + nl2·(u/H) − nl3·(u/H)²)`:nl3 = 奇次 S 型壓縮(主導,
  來源:PD 角度響應飽和/模糊圈超出相關窗)、nl2 = 偶次不對稱(球差)。
  測試釘死奇對稱性(disp(−u) == −disp(+u) 逐 bit)與端點壓縮比;
  ΔDCC 對合焦偏移:nl2 → 線性反對稱、nl3 → 對稱偶函數(兩者對 OQ#3 判讀意義不同)。
- **err(交叉驗證)對隨機雜訊天生鈍感**:σ_b = k·σ_d/√n → σ=2 也只有 err~1.7%。
  err 的設計目的是抓「b 與 focus 峰值的系統性分歧」;Sim 原本兩者同源導致 err 結構性≈0,
  新增 `focus_peak_offset` 注入後 `err = |offset|/span` 精確成立(offset 96 → 0.20 踩線)。
  → 若有人再問「為何雜訊拉滿不會 FAIL」,答案在此,不是 bug。

### 3.4 實作陷阱(踩過的坑,換手者注意)
- **ImPlot heatmap 第一列畫在最上方**。曾因多做一次列翻轉造成格值上下鏡射
  (DCC map 對稱看不出、err map 被使用者查核抓到)。現行約定:資料 row-major 直接餵,
  r=0 即畫面上排;點擊換算 `row = 5 − floor(y)`。**勿再翻轉**。
- **新版 macOS 無 /System/Library/Fonts/PingFang.ttc**(移入資產庫)→ 退到 Hiragino Sans GB
  (簡中),而它**缺 U+00B2(²)** → 「r?」顯示問題。解法:主 CJK 字型 + MergeMode 從
  Helvetica(Latin-1 補充)與 Apple Symbols(箭頭/數學/幾何)補 glyph;
  另 ChineseFull ranges 不含希臘字母/箭頭/幾何,需 builder 擴充。
  **GUI 首幀有 glyph 自檢**,缺字以 U+xxxx 記入 log。
- **C++ raw string 內含 `)"` 會提前終結字面值**(config 內建 JSON 的 `off)"`)
  → 一律用 `R"json(...)json"` 帶分隔符。
- AutoFit 與互動衝突:RAW 檢視器不可開 AutoFit(會鎖死縮放);資料圖開 AutoFit + 全域
  FitPadding 10%。heatmap 軸鎖定。
- `load_config` 曾漏驗 margin/tolerance 範圍(sweep.plan 層才擋)——config 負向測試抓到後補上。
  原則:**FR-01 的「載入即拒絕」必須在 loader 完成,不能依賴下游**。

## 5. 介面契約摘要(跨團隊,與 specs 同步)

- **disp_seq.json v1**(SPEC-004 §3a):自述式 grid(可 ≠ 8×6,D-5 聚合 median/quality 加權)、
  unit(raw_pixel|pd_image_grid,讀取端唯一入向轉換點)、dacs 嚴格遞增與 C-1 規劃 ±1 比對、
  **focus 必填**、null=無效。**尚未與外部 SAD 團隊正式凍結——M2 前必辦**。
- 外部 SAD 精度驗收:|bias| ≤ 0.3、σ ≤ 0.5 raw px(SPEC-005 §3a)。
- 單位契約:core 只有 raw_pixel;轉換僅讀取端/打包端兩處;report 雙單位 + ratio_check。
- RAW:唯讀、UI 底圖、缺檔降級(FR-18)。

## 6. 換手指南(新機器上手)

```bash
git clone git@github.com:huntkao/DCC_calibration.git && cd DCC_calibration
cmake -S . -B build -G Ninja      # 首次需網路(FetchContent 鎖版下載)
cmake --build build
ctest --test-dir build            # 應 56/56 綠
./build/src/dcc_cli/dcc_cal --dry-run --out /tmp/dcc_out       # CLI 驗證
./build/src/dcc_gui/dcc_gui --smoke                            # GUI 煙霧測試
./build/src/dcc_gui/dcc_gui                                    # 桌面工作台
```

- 閱讀順序:CLAUDE.md → 本文件 → specs/(SPEC-004 單位權威最重要)→ M0 階段報告(docs/)。
- GUI 首啟自動排版(選單「檢視→自動排版」可重套);session.json 可存讀工作現場。
- 改規格 = 改 specs/ 對應檔 + 頂部 rev 一行;commit 慣例 `M1:`/`spec:` 前綴。
- 每次改動的最低驗證:build 零警告(本專案 targets)→ ctest 全綠 → `dcc_gui --smoke`。
