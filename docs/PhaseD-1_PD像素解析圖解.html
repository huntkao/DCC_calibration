圖解:PD 像素解析與 L/R 影像尺寸推導

修正並展開 Phase D 第 1 步的計算 · 假設模組:4608×3456 sensor,32×32 pattern,每 pattern 8 對 L/R

勘誤:前版文件寫「L/R 影像各約 144 × 216」與「每區約 18 × 36 個 PD 樣本」——兩者皆誤,只算到正確值的 1/4。正確值為 288 × 432 與 36 × 72,推導如下。

STEP 1全幅影像切成多少個 pattern block?

橫向 block 數 = 4608 / 32 = 144 縱向 block 數 = 3456 / 32 = 108 全幅 block 總數 = 144 × 108 = 15,552 blocks

STEP 2單一 32×32 block 內,8 對 L/R 怎麼排?

假設 8 對以 2 欄 × 4 列配置——即每一對 L/R 佔據一個 16 × 8(寬×高)的子格。這是本文的關鍵假設,實際位置以 sensor 廠的 configuration file 為準,但「均勻子格」是最常見的設計。

L 像素總數 = 15,552 blocks × 8 = 124,416(R 相同) PD 像素密度 = (124,416 + 124,416) / (4608 × 3456) = 248,832 / 15,925,248 ≈ 1.56% ← sparse PDAF 典型範圍(1–3%)

STEP 3抽出後的 L / R 影像是多大?

把全幅所有 L 像素依原始相對位置collect成一張影像:水平方向每 16 px 有一個樣本、垂直方向每 8 px 有一個樣本,所以——

L 影像寬 = 4608 / 16 = 288 L 影像高 = 3456 / 8 = 432 L 影像 = 288 × 432(寬 × 高),R 影像相同 驗算:288 × 432 = 124,416 ✓ 與 STEP 2 的像素總數一致
單位陷阱(演算法開發必讀):SAD 位移搜尋是在 L/R 影像(288×432)的座標系做的。若搜尋結果為 d 格(次像素內插後),換算回原始 pixel 單位是 d × 16 px。DCC 的分母「disparity [pixel]」必須全流程統一單位——校正 DLL 用哪個座標系,runtime PDLIB 就得用同一個,否則 DCC 直接差 16 倍,一步跳焦會嚴重過衝或不動。

STEP 4每個 8×6 區域裡有多少 PD 樣本?

區域尺寸(原始座標)= 4608/8 × 3456/6 = 576 × 576 px 每區 L 樣本 = 576/16(橫)× 576/8(縱)= 36 × 72 = 2,592 個 (R 樣本同為 2,592 個) 驗算:48 區 × 2,592 = 124,416 ✓ 與 L 像素總數一致
